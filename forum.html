Forum

Go SQLite JavaScript Docker HTTP Auth View on GitHub

Overview

I built Forum as my deep dive into web fundamentals before I was ready to reach for frameworks. The goal was simple: understand what frameworks are actually doing for me before letting them do it. That meant writing everything from scratch — the HTTP server, the auth system, the database layer, and even the single-page app architecture on the frontend — using only the standard library on both ends.

The result is a Reddit-style discussion platform: users register, create posts under categories, comment, and like or dislike content. Filtering by category, by your own posts, or by posts you've liked is all handled without a page reload. It's containerised with Docker and backed by SQLite.

Backend: HTTP & Routing in Go

The entire backend runs on Go's net/http package — no Gin, no Echo, no router library. Writing raw HTTP handlers forces you to think carefully about things that frameworks handle silently: method checking, path parsing, middleware chaining, consistent error responses, and proper HTTP status codes. Every 404, 405, and 500 is deliberate and handled explicitly.

Middleware for authentication — checking that a valid session cookie exists before serving protected routes — is written as a plain handler wrapper. Once I built it myself, every authentication middleware I've used in frameworks since has made immediate sense.

Authentication: Sessions, Cookies & Bcrypt

Auth was the most instructive part of the project. The full lifecycle: a user registers with an email and password. The password is hashed with bcrypt before touching the database — never stored in plaintext. On login, the server generates a UUID session token, stores it in SQLite alongside the user ID and an expiration timestamp, then sends it to the client as an HttpOnly cookie.

On every subsequent protected request, the server reads the cookie, looks up the token in the sessions table, checks expiry, and either proceeds or redirects to login. Only one active session is allowed per user — logging in from a new device invalidates the previous session. Building this by hand made JWT, OAuth, and session libraries genuinely transparent to me; I know exactly what problem they're solving.

Database: SQLite & Schema Design

The data layer is SQLite accessed through Go's database/sql interface with raw queries — no ORM. I designed the schema from scratch using entity relationship diagrams before writing a line of SQL: users, posts, categories, comments, likes, and sessions all needed to relate correctly, and getting the foreign key structure right upfront saved significant pain later.

Writing raw SQL — joins across four tables to render a post with its comment count, category, and like/dislike totals — taught me things about query planning and indexing that an ORM would have hidden. It also forced me to handle SQL errors explicitly at every call site rather than having them silently swallowed.

Frontend: A SPA Without a Framework

The frontend is the part I'm most proud of. It's a genuine single-page application — one HTML shell, client-side routing, and dynamic rendering — built entirely in vanilla JavaScript. No React, no Vue, no module bundler. Just the DOM API and fetch.

Client-side routing intercepts link clicks and popstate events, updates the URL with the History API, then fetches data and re-renders the relevant section of the page. Managing UI state by hand — knowing when to invalidate a cached view, how to handle loading and error states, how to keep the URL and the rendered content in sync — gave me a concrete understanding of what React's component model, state management, and reconciler are actually doing underneath the abstraction.

Challenges

The trickiest bugs were around concurrency. Go's HTTP server handles each request in its own goroutine, which means multiple requests can hit the database simultaneously. SQLite in WAL mode handles concurrent reads fine, but write contention required careful transaction scoping to avoid data races — particularly around the like/dislike counters where two users could update the same row at the same moment.

Cookie scoping was another source of bugs. Getting the path, domain, and SameSite attributes right so that cookies are sent on exactly the requests you expect — and not on others — took more iteration than I anticipated. Session expiry edge cases (expired token present, token missing, token valid but user deleted) each needed distinct handling to avoid confusing redirects.

On the frontend, the hardest problem was preventing XSS. User-generated content rendered into the DOM needs to be escaped every time — using textContent instead of innerHTML where possible, and sanitising carefully where HTML rendering is unavoidable.

What I Took Away

Building a full web application without any framework — from the TCP socket up to the DOM — is the most educational thing I've done as a developer. Every abstraction I use now, I understand. I know what a session middleware is doing, I know why React exists, I know what an ORM is hiding. That foundation changes how I approach every project that comes after it.
